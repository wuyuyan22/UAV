# 机间 Mesh 自组网扩展实现方案

| 项目 | 说明 |
|------|------|
| 文档版本 | v0.1 |

| 定位 | **在 5G 主方案之上的二期/并行扩展**：机间链路从「预配置星型」演进为「可自愈、可多跳的 Mesh 承载」；**不替代** 5G 主方案中 GCS 每机直连、图传分端口、QoS、健康监测等结论，与之**正交组合**。 |

## 1. 对 5G 主方案的简要审视与后续优化方向

### 1.1 主方案已较好覆盖的部分（建议保持）

- **业务面拆分**：机间（19870）、GCS（14550）、图传（独立端口）三平面隔离。
- **每机直连 MP + 会话表**：避免长机合成遥测的妥协，与 Mesh **无冲突**（Mesh 主要改机间面）。
- **VPN + QoS + 5G 健康监测 + 降级**：属于接入网与可靠性，Mesh 层应**复用**而非重写。
- **`ISwarmAirLink` / `LinkPacket` 抽象**：为替换承载层提供清晰边界，**利于**下沉 Mesh。

### 1.2 主方案在「机间拓扑」上的隐含约束（Mesh 要解决的）

| 点 | v1.2 现状 | 风险或成本 |
|----|-----------|------------|
| 机间寻址 | 僚机 `--leader-ip` 固定指向长机 VPN IP | 长机伴飞故障或 IP 变更时，**全网需改配置或重启**；无中继。 |
| 拓扑语义 | 「机间 UDP 汇聚至长机」星型 | 僚机–僚机不直连；**单点**与跳数=1 假设强绑定代码分支（`receive_loop` 长/僚分离）。 |
| 扩展性 | 文档已指出多机受 MP/入口带宽约束 | Mesh 后机间可绕开部分单跳瓶颈，但 **GCS 侧聚合**仍是独立瓶颈。 |
| 文档内可澄清项 | 长机 `InterUavLink` 仅 `bind`、未在 `init` 中 `set_target` 的路径 | 与「向多僚机单播/组播」的部署假设应对齐代码与运维说明，避免现场歧义。 |

### 1.3 建议的「主方案」小步优化（与 Mesh 可并行）

1. **机间 CLI 增强**：`--leader-sysid`、`--mesh-mode` 预留；长机侧显式 **`--inter-uav-bind-only` / `--inter-uav-send-broadcast`** 等与实现一致的参数文档化。  
2. **配置下发**：将 `leader_ip`、`followers` 等从静态文件扩展为 **地面可发 MAVLink 用户命令 + 本地持久化**（与既有群控 `COMMAND_LONG` 扩展衔接）。  
3. **观测**：机间增加 **对端 RTT、丢包、邻居表版本** 的 `STATUSTEXT` 或 MAVLink 调试消息，便于 Mesh 前后对比。  
4. **安全**：主方案已写一期依赖 VPN；Mesh 若引入 **机间直连 Wi‑Fi**，需单独写 **组密钥 / 帧计数器**，不可仅依赖蜂窝 VPN。

## 2. Mesh 扩展的目标与非目标

### 2.1 建设目标

1. **机间承载**：在保持 **`LinkPacket` 语义与 `ISwarmAirLink` 接口** 的前提下，使机间包可经 **多跳** 到达 `dst_id` 对应 sysid；**单跳星型** 成为 Mesh 的退化配置。  
2. **邻居与可达性**：节点自动维护 **「sysid ↔ 下一跳 / 链路质量」**；拓扑变化时 **秒级～十秒级** 收敛（指标可按机型裁剪）。  
3. **与逻辑长机编队兼容（一期）**：**编队数学与 `STATE_REPORT` / `FOLLOWER_CMD` 仍以逻辑长机为中心**；Mesh 只解决「包怎么到」，不强制一期做去中心编队。  
4. **与 5G 关系**：Mesh 可运行于 **(a)** 同一 VPN 上的用户态路由，或 **(b)** 独立机间网卡（如 Wi‑Fi 802.11s）+ 策略路由；**GCS 仍默认走 5G+VPN 主方案**。

### 2.2 非目标（本 Mesh 文档不展开或排期靠后）

- 替代 MP 多机会话表与图传 pipeline。  
- 完全去中心、无协调器的分布式编队一致性（CRDT/选举合并航迹）。  
- 运营商侧 5G 侧链（Sidelink）专有网规合规（需单独立项）。

## 3. 总体架构：双平面 + 可插拔 Mesh 承载

```
                    ┌─────────────────────────────────────┐
                    │  地面 MP（5G 主方案不变）              │
                    │  每机 GCS UDP / 会话表 / 多机 UI      │
                    └───────────▲────────────▲──────────────┘
                                │ VPN       │
        ┌───────────────────────┴───┐   ┌───┴───────────────────────┐
        │ 伴飞 #1  GcsIpLink        │   │ 伴飞 #N  GcsIpLink         │
        │ +「MeshAirLink」或内核路由 │   │ +「MeshAirLink」…          │
        └───────────┬───────────────┘   └───────────┬───────────────┘
                    │                                 │
                    │    机间 Mesh（本方案）           │
                    │    HELLO / 路由表 / 可选多跳转发  │
                    └─────────────────┬───────────────┘
                                      │
                              FC 串口 MAVLink（不变）
```

**原则**：

1. **控制面 / 数据面**：Mesh 控制面（邻居、路由）与 **`LinkPacket` 业务面** 分离端口或分离线程，避免 HELLO 洪泛影响 19870 时序。  
2. **向后兼容**：`--mesh-mode off` 时行为与 5G 主方案 **完全一致**（`InterUavLink` 直连长机 IP）。  
3. **长机逻辑长机**：一期 **不强制** 改变 `SwarmController` 中长机 `run_leader_tick` 与僚机 `run_follower_tick` 的划分；仅将 `air_link_->send_packet` 从「单下一跳 UDP」变为「查表发送」。

## 4. 分层设计

### 4.1 物理与链路层（选型）

| 选项 | 适用 | 说明 |
|------|------|------|
| **A. VPN + 用户态 Mesh** | 最快与主方案拼接 | 不改变空口；在每机跑 **路由/转发**（见 §5），仍用现有 UDP socket。 |
| **B. Wi‑Fi 802.11s / 自研跳频** | 机间卸载蜂窝 | 与 5G 并行接口；Linux 上 **batman-adv / OLSR** 提供 L3 可达，伴飞进程 **只绑 mesh 接口 IP**。 |
| **C. 混合** | 外场推荐中长期 | 5G 保 GCS/图传；**机间专网 Mesh** 降时延与蜂窝切换抖动对编队的影响。 |

一期建议 **A 或 C 的「A 阶段」**：验证多跳与选路，再引入 B。

### 4.2 网络层（路由）

- **小规模（≤8 机）**：自研 **链路状态（LS）** 或 **距离向量（DV）** + 序列号防环 + **ETX 权重**；周期 **200–1000 ms** 交换摘要。  
- **中大规模或运维优先**：**BATMAN-adv**（内核模块）或 **OLSRv2**，伴飞仅使用 **已路由的 IP** 调用现有 `send_to`。  
- **与 sysid 映射**：维护表 **`sysid → primary_ip`（及可选 secondary）**；HELLO 中带 **VPN IP + mesh IP + sysid**。

### 4.3 传输与业务（现有代码）

- **`MeshAirLink : public ISwarmAirLink`**：  
  - `send_packet(pkt)`：根据 `pkt.dst_id` 查路由 → `send_to(next_hop, inter_uav_port)`；若 `dst_id` 为本机 sysid 则交本地虚拟环回。  
  - `recv_packet`：非本机目的且 **TTL>0** 时 **转发**（用户态中继）或交给内核（若用 batman）。  
- **`SwarmController`**：`init` 中 `air_link_ = &mesh_air_link_` 或 `&link_udp_` 由配置决定；**尽量减少** 对 `receive_loop` 内角色分支的侵入（二期再抽象「机间入口统一分发」）。

### 4.4 应用与选主（可选二期）

- **逻辑长机故障**：由 **地面指令** 或 **机间超时 + 最低 sysid 获胜** 触发 **「协调器迁移」**；与《5G 主方案》中「进程级重启」策略衔接，见独立「领导权移交」设计文档。  
- Mesh 层只保证 **新协调器 IP 可达**，不强制改变编队算法。

## 5. 一期推荐实现路径（分阶段）

### 阶段 0：静态多跳（1 周级 PoC）

- 3 机链：**A—B—C**，仅 B 开启 **用户态转发**（收到 `dst=C` 且非本机则 `send_to(C)`）。  
- 路由表 **写死** 或 JSON 配置；**不改** `LinkPacket` 编解码。  
- 验收：C 上 `STATE_REPORT` 可达 A，`FOLLOWER_CMD` A→C 经 B 往返时延可测。

### 阶段 1：HELLO + sysid↔IP + 超时剔除

- 新增 UDP 端口 **`19871`（示例）** 或 **组播地址** 发 **HELLO**（`sysid`、序列、可选 RSSI）。  
- 每机构建 **邻居表**；`MeshAirLink` 根据表计算 **下一跳**（仍假设 **逻辑星型到长机** 时，可简化为「到长机 sysid 的下一跳」）。  
- 验收：拔掉中间某链路后，**备用路径**（若存在）在 T 秒内恢复或上报 `STATUSTEXT`。

### 阶段 2：与 5G 主方案联调

- **策略路由**：`10.8.0.0/24` 上 GCS 仍走 `wg0`；若 mesh 用 `wlan0` 另一网段，机间目的走 **mesh 路由表**。  
- **QoS**：Mesh 控制面小包 **DSCP CS6**；`LinkPacket` 业务与现有一致 **AF41** 或与主方案 §6.5 对齐。  
- **健康监测**：`5GHealthMonitor` 只管 **GcsIpLink**；Mesh 链路上报 **独立指标**（丢包、邻居数），避免误判蜂窝掉线为 Mesh 故障。

### 阶段 3：内核路由或 BATMAN（可选）

- 将用户态转发迁入 **batman-adv**，减少 CPU 与拷贝；伴飞配置 **`--inter-uav-iface bat0`**。

## 6. 协议与消息建议（机间控制面）

与 `LinkMsgType`（`inter_uav_link.h`）**并列**，建议 **新端口** 承载控制面，避免与现有「用 GLOBAL_POSITION_INT 伪装 STATE_REPORT」的线格式混排误判：

| 消息 | 方向 | 载荷要点 |
|------|------|------------|
| MESH_HELLO | 周期性广播/组播 | `sysid`, `seq`, `vpn_ip`, `uptime_s` |
| MESH_LS_U | 单播或可靠洪泛 | 链路状态摘要（邻居列表 + ETX） |
| MESH_ROUTE_REQ / REP | 按需 | 类似 AODV 的可选优化，一期可省略 |

**安全**：HELLO 带 **HMAC-SHA256(sysid, seq, key)** 或 TLS 仅用于控制面小通道；**业务 `LinkPacket` 线格式** 一期可仍依赖 VPN；若 mesh 为开放 Wi‑Fi，则 **必须** 加密或签帧。

| 级别 | 场景 | 指标示例 |

| T0 | 星型退化 | 与现网一致，编队丢包率、时延不降 |

## 9. 风险与缓解

| 风险 | 缓解 |
|------|------|
| 广播风暴 | HELLO 限速、随机退避、跳数上限 |
| 路由环路 | 序列号、TTL、或仅用 DAG 到协调器 |
| 与 MAVLink 机间伪装帧混淆 | 控制面独立端口 / 显式新 msgid 范围 |
| 双协调器 | 地面授权 + 任期号；或进程级互斥 |
| 嵌入式 CPU | 优先内核路由；HELLO 合并、降低频率 |

---

## 10. 结论

- **5G 主方案**继续作为 **蜂窝接入、GCS 直连、图传、QoS、健康监测** 的基线；其机间「长机 IP 星型」在 **规模化与容错** 上存在自然上限。  
- **本 Mesh 方案**在 **不改变 `LinkPacket` 编队语义** 的前提下，通过 **`MeshAirLink` + 邻居发现 + 路由（用户态或内核）** 扩展机间可达性，**分阶段**从静态中继演进到自组织。  
- **文档维护**：主方案 v1.3+ 可增加一节「机间拓扑演进」指向本文；本文版本随实现迭代 `v0.1 → v1.0`。
