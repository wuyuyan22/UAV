# 鲁班猫0 对接 ArduPilot 多旋翼与群控系统方案（优化版）

## 一、文档目标与适用范围

本文档面向 `UAV_Swarm_APM` 项目，目标是建立一套可落地的“伴机控制 + 群控接入 + 地面站协同”方案：

- 单机层：鲁班猫0 对接 ArduPilot 飞控，完成 Guided 控制闭环；
- 群控层：Leader/Follower 协同与机间链路控制；
- 地面站层：Mission Planner 负责任务编排、批量下发和状态监控。

> 核心原则：高频控制闭环在伴机侧完成，地面站只做编排与监控，不参与高频姿态/位置闭环。

## 三、硬件与链路建议

### 3.1 鲁班猫0 到飞控

- 首选串口直连：`/dev/ttyS8`（UART8）；
- 备选 USB：`/dev/ttyACM0` 或 `/dev/ttyUSB0`；
- 波特率建议 `57600` 或 `115200`，需与飞控 `SERIALx_BAUD` 一致。

### 3.2 飞控关键参数

- `SERIAL1_PROTOCOL = 2`（MAVLink2）；
- `SERIAL1_BAUD = 57/115`（分别对应 57600/115200）；
- `GUID_TIMEOUT = 3`（建议保持默认，伴机需持续下发目标）；
- 建议打开合适的失效保护参数（按机型和实飞策略配置）。

### 3.3 机间通信

- Leader/Follower 建议使用局域网 UDP（网线优先于 WiFi）；
- 建议固定 IP 并建立 `sysid -> ip:port` 映射，避免目标路由不确定。

### 3.4 P900 电台通信方案（当前拟采用）

若项目计划采用 P900 作为主通信链路，建议按“分层通信”设计：

- 地面站 <-> Leader：P900 主链路（任务下发、群控命令、关键状态回传）；
- Leader <-> Follower：
  - 推荐：近距 UDP（WiFi/网线）跑高频协同，P900 做兜底命令链路；
  - 备选：全 P900，但必须降频并精简消息字段。

约束原则：

- P900 适合低速高可靠命令，不适合高频大带宽全量遥测；
- 群控高频闭环必须保留在伴机本地，不依赖地面站持续点控。

## 四、协议与命令设计（与当前项目对齐）

## 4.1 飞控控制协议

- 伴机与飞控：MAVLink；
- Guided 控制主消息：
  - `SET_POSITION_TARGET_GLOBAL_INT`
  - `SET_POSITION_TARGET_LOCAL_NED`
  - `SET_ATTITUDE_TARGET`（按需求）

硬约束：Guided 目标下发建议 `>=2Hz`，否则 `GUID_TIMEOUT` 后飞控进入保护行为。

## 4.2 群控控制命令（GCS -> swarm_node）

使用 `COMMAND_LONG` + `MAV_CMD_USER_1..5`：

- `USER_1`：开始编队；
- `USER_2`：停止编队；
- `USER_3`：RTL；
- `USER_4`：LAND；
- `USER_5`：ARM + TAKEOFF（`param1=高度`）。

路由约束：

- `target_system = swarm_node sysid`；
- `target_component = MAV_COMP_ID_ONBOARD_COMPUTER`（或 `MAV_COMP_ID_ALL`）。

## 4.3 机间消息（Leader <-> Follower）

- `STATE_REPORT`：follower 上报状态；
- `FOLLOWER_CMD`：leader 下发目标；
- 当前实现基于 MAVLink 标准消息承载，便于调试与扩展。

## 4.4 P900 链路下的消息分级与频率建议

消息优先级（高到低）：

1. 安全控制：`STOP`/`RTL`/`LAND`/急停；
2. 阶段控制：起飞、开始编队、任务阶段切换；
3. 状态摘要：模式、定位健康、电池、关键告警；
4. 低优先调试：详细遥测、日志。

建议频率基线（初始值）：

- GCS -> Leader 控制命令：`1~2 Hz`（事件触发优先）；
- Leader -> Follower 目标下发（全 P900 场景）：`2~5 Hz`；
- Follower -> Leader 状态回传：`1~2 Hz` + 异常事件立即上报。

可靠性要求：

- 全部控制命令必须带 `seq`；
- 必须实现 ACK/NACK + 超时重试；
- 命令处理必须幂等（重复包不产生副作用）。

## 五、控制流程（单机与群控）

## 5.1 单机最小闭环

1. 建立 MAVLink；
2. 等待心跳；
3. 切 `GUIDED`；
4. ARM；
5. TAKEOFF；
6. 周期下发目标；
7. LAND/RTL。

## 5.2 群控任务闭环

1. 地面站选择机群并校验在线状态；
2. 上传任务（Leader 或全体）；
3. 统一发起 `ARM+TAKEOFF`；
4. 统一发 `START_FORMATION`；
5. 飞行中持续监控与参数调整；
6. 结束阶段发 `STOP_FORMATION` 或 `RTL/LAND`。

## 六、结合当前项目的优化项（优先级）

## P0：必须先做

1. Leader 下发路由显式化  
   增加 `sysid -> ip:port` 路由表或命令行配置，避免“发送成功但 follower 收不到”。

2. 任务入口补齐  
   增加 `--wp-file`、`--auto-start`，形成可复现任务闭环。

3. 命令回执机制  
   每条群控命令需要 ACK/NACK（含错误码），便于地面站重试与告警。

## P1：稳定性增强

1. 并发访问收敛  
   减少裸指针跨线程使用，统一快照读取状态。

2. 可观测性增强  
   增加指标：目标下发频率、链路包龄、编队误差、丢包率。

3. 机型兼容一致性  
   Copter/Plane 分支在 Guided 保活和模式判断上分别处理。

## P2：工程化

1. SITL 回归矩阵（2/3/5 机规模）；
2. 异常场景测试（掉线、延迟、单机故障）；
3. 参数在线调整与持久化。

## 七、Mission Planner 接入建议（群控地面站化）

## 7.1 接入定位

- 地面站不做编队闭环算法；
- 地面站负责“任务编排 + 批量下发 + 状态监控 + 异常处置”。

## 7.2 最小可用功能（MVP）

1. 机群管理：维护 `sysid` 列表与角色；
2. 批量命令：一键起飞/开始编队/停止/RTL/LAND；
3. 任务下发：先支持 Leader 任务上传；
4. 状态看板：在线状态、模式、电池、任务阶段。

## 7.3 可靠性策略

- 命令超时重试；
- 事务化执行（预检查 -> 下发 -> ACK 校验 -> 进入下一阶段）；
- 失败回滚（部分失败触发 STOP 或 RTL 策略）。
- P900 场景下需额外启用限流与优先级调度，防止低优先遥测挤占控制通道。

## 7.4 P900 组网建议

- 拓扑建议：`地面站 -> Leader -> Follower`，避免所有节点同频全互联竞争；
- 使用固定地址与节点白名单，减少串扰；
- 为关键命令预留最小带宽配额（不可被普通遥测占满）；
- 对链路质量建立监控：RSSI、丢包率、往返时延、最近 ACK 时间。

## 八、推荐实施路径

## 阶段 1（1-2 周）

- 打通单机 Guided 闭环；
- 修复 Leader 路由；
- 完成多机 SITL 基础联调。

## 阶段 2（1-2 周）

- 接入 Mission Planner 的群控命令面板（MVP）；
- 加入命令 ACK、重试与日志。
- 完成 P900 链路参数整定（频率、重试、超时、队列阈值）。

## 阶段 3（持续）

- 增强任务编排、参数在线调整；
- 做回归测试与实飞验证标准化；
- 完成“链路退化降级”状态机联调：`NORMAL -> DEGRADED -> SAFE_RTL`。

---

## 九、实飞安全清单（执行前必查）

- 桨叶与动力安全隔离；
- 地理围栏与失效保护参数已验证；
- 指令频率/模式切换日志可见；
- 所有节点时钟与网络稳定；
- 异常时可一键 RTL/急停流程明确。

---

## 十、结论

鲁班猫0 + ArduPilot 的技术路线正确，`UAV_Swarm_APM` 已具备群控基础。  
要从“可跑”走向“可用”，重点不是继续堆控制算法，而是优先补齐：**路由确定性、任务闭环、命令回执、可观测性**。  
完成这些后，再接入 Mission Planner 形成地面站群控能力，整体系统即可进入稳定迭代阶段。
