# UAV_BZ_L 通信链路详细分析文档

## 1. 文档目标与范围

本文档用于回答以下问题：

- 地面站与无人机（尤其长机）之间如何通信；
- 长机与僚机之间如何通信；
- 两者是否同一条物理链路；
- 遥测数据（位置、姿态、健康、指令等）如何采集、汇总、上报；
- 现场如何排查链路拥塞。

分析基于当前项目实现（`IET_UAV_BZ_L`），重点关注 `Node.cpp`、`Node.h`、`test.cpp`。

---

## 2. 总体结论

- 当前主流程并非 MAVLink，而是**自定义二进制协议**（帧头 `0xFF 0xFA` + 长度 + 序号 + 地址/组件 + 类型 + CRC16）。
- **长机 <-> 地面站** 与 **长机 <-> 僚机** 在当前启用实现中主要复用同一个串口：`fdSerial_422L`（`/dev/COM422_L`，115200）。
- **长机 <-> 本机飞控 CHS40** 使用另一条独立串口：`fdSerial_232`（`/dev/COM232`）。
- UDP/TCP 代码存在，但当前 `main` 主流程中并未作为核心链路启用，属于历史/预留通道。

---

## 3. 物理链路与接口映射

### 3.1 串口资源

- `fdSerial_232`：`/dev/COM232`
  - 用途：长机与 CHS40 飞控交互（读取飞控状态、下发航点/切线等）。
- `fdSerial_422L`：`/dev/COM422_L`
  - 用途：复用承载
    - 地面站命令下发；
    - 长机对地状态上报与 ACK；
    - 僚机状态回传；
    - 长机对僚机指令转发。

### 3.2 网络接口（非当前主链路）

- UDP 9876/3326、TCP server/client 相关函数存在；
- 但在 `test.cpp` 主流程中未启动对应线程，故当前部署重点仍是串口链路。

---

## 4. 协议结构（同口复用的关键）

项目核心协议通过包内字段区分“谁发给谁”，而不是靠不同物理口。

## 4.1 标准帧结构（`0xFF 0xFA` 协议）

- `[0]` 帧头1：`0xFF`
- `[1]` 帧头2：`0xFA`
- `[2..3]` 长度 `len`
- `[4..7]` 包序号 `seq`
- `[8]` 源节点 `source_id`
- `[9]` 目的节点 `dest_id`
- `[10]` 源组件 `source_component_id`
- `[11]` 目的组件 `dest_component_id`
- `[12]` 消息类型 `msg_type`
- `[13..]` 载荷 `payload`
- 末尾 2 字节 CRC16（多项式 `0x8005`）

### 4.2 解析入口

- 接收统一进入 `package_parse()`；
- 当 `dest_id == 本机id` 时，再按 `msg_type` 分发处理；
- 这就是“同口复用”的协议路由核心。

---

## 5. 遥测数据上报链路（位置等信息如何到地面站）

## 5.1 数据源：本机飞控

- `read_s40()` 从 `fdSerial_232` 读取 CHS40 数据；
- `Parse_CHS40_data()` 解析帧后调用：
  - `decode_frame1()`：姿态/高度/速度相关；
  - `decode_frame2()`：GPS状态、飞行模式、健康位；
  - `decode_frame3()`：经纬度、航点索引、时间等；
- 解析结果写入 `unit.state`，并更新 `group.UpdateUnit(unit.state)`。

## 5.2 数据源：僚机状态

- `read_p840()` 从 `fdSerial_422L` 读数据；
- 进入 `package_parse()`；
- 当 `msg_type==0x01` 且 `dest_component_id==0x03` 时，执行 `decode_uav_state_p840()`；
- 解码后更新 `group` 中对应僚机状态。

## 5.3 汇总与上报

- 长机周期性调用 `send_uav_state_to_gcs()`；
- 构造 `msg_type=0x0A` 状态包，目标 `dest_id=0x0C`（地面站）；
- 从 `group.unit_list` 遍历写入所有机体状态（每机40字节）；
- 通过 `sendDataTty(fdSerial_422L, ...)` 发出。

## 5.4 发送频率

`test.cpp` 中 `node_run` 每 100ms 执行一次：

- `counter % 10 == 0 && group_role==LEADER` 时发送地面站状态；
- 即地面站状态上报约 **1Hz**。

---

## 6. 主要消息类型与方向

以下为当前主链路上典型消息：

- `0x01`：无人机状态（僚机 -> 长机，或节点状态类消息）
- `0x02`：航点下发（地面站 -> 长机）
- `0x03`：开始编队（地面站 -> 长机 -> 僚机控制）
- `0x04`：结束/暂停编队（地面站 -> 长机 -> 僚机控制）
- `0x05`：结束/重启类控制（项目中有系统重启逻辑）
- `0x06`：长机对僚机的航点/协同控制数据
- `0x07`：编队参数变更（间距、角度、速度/高度限制等）
- `0x0A`：地面站状态上报（长机 -> 地面站）
- `0x0B/0x0C/0x0D/0x0E/0xAD/0xAE/0xBA/0xBB...`：ACK/状态反馈
- `0xAB`：航点回传（长机 -> 地面站）
- `0xAC`：编队参数回执（长机 -> 地面站）
- `0xCF`：控制频率回执（长机 -> 地面站）

## 8. 长机是否在做“网关”

是。长机节点在当前实现中承担了明显网关职责：

- 向下采集：飞控数据（232）+ 僚机数据（422）；
- 向上汇总：统一封装后上报地面站（422）；
- 横向转发：将地面站命令转换为僚机/飞控可执行格式并发送。

这意味着长机链路质量会直接决定整个编队的信息通达性。

## 9. 拥塞风险评估

## 9.1 风险来源

- 单口复用：地面站下发、僚机上报、长机回传、ACK 全部抢占 `RS422_L`；
- 周期任务叠加：100ms 主循环内可能同时触发多个发送函数；
- 突发大包：航点回传（`0xAB`）和多机状态聚合（`0x0A`）可能形成突发；
- 重发放大：若上层超时重发策略激进，容易形成 ACK/重发风暴；
- 缺乏精细队列：当前代码多为“到点即发”，未见统一发送整形调度。

## 9.2 典型现场现象

- 地面站偶发“跳点”或状态延迟；
- 僚机连接状态位间歇掉线；
- 参数下发后回执不稳定；
- 一旦执行批量航点操作，短时控制响应变慢。

## 10. 排障建议（按优先级）

### 10.1 先做观测

- 给 `sendDataTty(fdSerial_422L, ...)` 增加最小日志：
  - 时间戳、消息类型、字节数、发送来源函数；
- 给 `read_p840()` 增加统计：
  - 每秒接收字节数、包数、CRC失败数、按类型计数。

### 10.2 再做限流

- 对低优先级回报降频（例如状态包从 1Hz 进一步按场景动态调整）；
- ACK 合并或限速，避免连续触发；
- 航点/参数批量回传时，增加发送间隔（小包化+节流）。

### 10.3 最后做架构优化

- 若硬件允许，拆分链路：
  - 地面站独立口；
  - 机间链路独立口；
- 或提升波特率并引入发送队列（优先级调度：控制 > 状态 > 回传）。

---

## 11. 可执行的带宽预算方法（建议现场使用）

按以下方式做快速预算：

`链路占用(bps) = Σ(每类消息字节数 * 10bit * 频率Hz)`

说明：

- 串口按 8N1 估算时，1 字节约 10bit（含起止位）；
- 若总占用长期超过链路能力的 60%~70%，拥塞风险显著上升；
- 需要把“周期流量 + 突发流量 + 重发流量”一起算。

建议现场至少统计：

- `0x0A` 状态包平均字节数与频率；
- `0x06` 僚机控制包频率；
- ACK 包数量；
- CRC错误率与超时重发次数。

---

## 12. 关键结论回顾

- 长机到地面站与长机到僚机：**当前实现主要同一物理链路（RS422_L）复用**；
- 长机到飞控：**独立 COM232**，不在该复用口上；
- 同口复用依赖地址字段与消息类型分流；
- 现场拥塞本质上是“共享口争用 + 突发发送 + 重发放大”；
- 优先从“统计观测 -> 限流整形 -> 物理拆分/提速”三步落地。
